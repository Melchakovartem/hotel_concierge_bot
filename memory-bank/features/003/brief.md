# Feature 003: Hotel staff ticket workflow without admin involvement

## 1. Проблема

Сейчас операционный сценарий обработки гостевого тикета, который уже создан гостем или системой внутри одного отеля, не может пройти от начала до конца без роли `admin`: операционные роли отеля не могут выполнить цепочку «`manager` создает staff-исполнителя для отеля → `manager` назначает тикет этому исполнителю → назначенный `staff` видит назначенный тикет → назначенный `staff` переводит тикет из `new` в `in_progress` → назначенный `staff` переводит тикет из `in_progress` в `done`». Метрика проблемы — 0 успешных end-to-end проходов этой цепочки операционными ролями без вмешательства `admin`.

Это блокирует end-to-end процесс обработки гостевого тикета внутри отеля:

* `manager` не может добавить `staff`-пользователя для работы с тикетами своего отеля;
* `manager` не может назначить тикет сотруднику без участия `admin`;
* `staff` не может принять назначенный ему тикет в работу и зафиксировать завершение работы.

В результате платформа поддерживает административную настройку, но не поддерживает полный операционный цикл обработки гостевого тикета персоналом отеля: назначение исполнителя, взятие тикета в работу и фиксацию завершения.

---

## 2. Стейкхолдер

Основные пользователи, для которых решается проблема, — `manager` и `staff` отеля.

`manager` отвечает за организацию операционной работы внутри отеля, к которому привязан его аккаунт: добавление сотрудников, распределение гостевых тикетов, контроль прогресса и завершение обработки запросов.

`staff` отвечает за выполнение гостевых тикетов в рамках одного отдела отеля, к которому привязан его аккаунт, и назначенных ему задач внутри этого отеля.

`admin` — это администратор платформы, а не сотрудник отеля. `admin` управляет настройкой платформы и не должен быть нужен для обработки гостевых тикетов внутри отеля.

---

## 3. Контекст

Эта фича продолжает Phase 1: `EPIC 1.2 — Staff & Departments` и `EPIC 1.3 — Ticket Management Core`.

В проекте уже есть базовый домен для операционной работы отеля: hotels, staff, departments, tickets, статусы тикетов и управление на уровне `admin`. Задача возникла после Feature 002, где появились admin-only Hotel CRUD и read-only hotel-scoped views для staff и tickets: при проверке Phase 1 стало видно, что основной end-to-end сценарий обработки гостевого тикета остается невыполнимым операционными ролями, потому что ключевые действия завязаны на `admin`.

Это важно сейчас, потому что продукт не может поддержать полный процесс обработки гостевого запроса внутри отеля, пока добавление исполнителя, назначение тикета и завершение работы зависят от администратора платформы.

---

## 4. Желаемый результат

Желаемый результат: операционные роли могут выполнить описанный сценарий обработки гостевого тикета, который уже создан гостем или системой в их отеле, без участия `admin`: `manager` создает staff-исполнителя и назначает ему тикет, а назначенный `staff` видит этот тикет, переводит его в работу и завершает обработку. Целевая метрика — минимум 1 успешный end-to-end проход этой цепочки операционными ролями без вмешательства `admin` в проверяемом сценарии фичи.

Результат считается достигнутым, когда:

* `manager` и `staff` работают только с данными своего отеля и не имеют доступа к данным других отелей;
* `manager` может создавать новых `staff`-пользователей для своего отеля;
* каждый новый `staff`-пользователь привязан к одному отделу внутри своего отеля;
* `manager` видит все тикеты своего отеля;
* `manager` может назначать тикеты пользователям роли `staff` из того же отеля;
* `manager` может менять назначенного исполнителя тикета и устанавливать один из существующих статусов тикета: `new`, `in_progress`, `done`, `canceled`; исходный текст гостевого запроса и гость, создавший тикет, не меняются;
* `staff` всегда принадлежит ровно одному отделу отеля; `staff` видит тикеты своего отеля, которые либо назначены ему лично, либо привязаны к тому же отделу, что и этот `staff`;
* `staff` может перевести назначенный ему тикет из `new` в `in_progress` и из `in_progress` в `done`;
* завершение обработки тикета в статусе `done` не требует действий роли `admin`;
* end-to-end сценарий обработки гостевого тикета, который уже создан гостем или системой (`manager` создает staff-исполнителя для отеля → `manager` назначает тикет этому исполнителю → назначенный `staff` видит назначенный тикет → назначенный `staff` переводит тикет в `in_progress` → назначенный `staff` переводит тикет в `done`), проходит без единого вмешательства `admin`.

## 5. Не входит в цель

Этот Brief не определяет техническую реализацию, структуру routing, UI-layout или классы authorization policies.

Следующие темы не входят в бизнес-цель этой фичи:

* универсальный RBAC или ACL permissions matrix;
* audit log;
* notification system;
* SLA tracking;
* bulk ticket operations;
* изменения, выходящие за пределы перечисленных выше ролей, данных и действий.
