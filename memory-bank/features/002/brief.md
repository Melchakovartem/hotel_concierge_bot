# Feature 002 — Role-based authorization & Hotel CRUD

## Задача

Добавить ролевую авторизацию для контроллеров `Staff` и `Tickets`, а также
реализовать полный CRUD для отелей (создание, редактирование, удаление).

## Проблема

После slice 001 аутентификация работает через Staff-записи, но ролевые проверки
отсутствуют — любой авторизованный сотрудник видит все страницы. Кроме того,
управление отелями ограничено только просмотром (`#index`).

## Контекст

Зависит от slice 001 (auth-паттерн в `BaseController`). Роли: `admin`,
`manager`, `staff`. Модель `Hotel` уже имеет `name` и `timezone` (оба
обязательные). Ассоциации отеля используют `dependent: :restrict_with_exception`.

## Для кого

Сотрудники с ролью `admin` — полный доступ. `Manager` — чтение тикетов.
`Staff` — ограниченный доступ или отказ в зависимости от контроллера.

## Ожидаемый результат

- `GET /admin/staff` — доступен только `admin`
- `GET /admin/tickets` — доступен `admin` и `manager`; поведение `staff` TBD
- `GET/POST /admin/hotels/new` — создание отеля (`admin` only), через form + service object
- `GET/PATCH /admin/hotels/:id/edit` — редактирование (`admin` only)
- `DELETE /admin/hotels/:id` — удаление с graceful error при наличии связанных записей

## Ограничения

- Не менять `BaseController` и auth-механизм (slice 001)
- Не трогать `dependent:` на ассоциациях Hotel
- Валидация только в form object, не в модели
- Без клиентской валидации и bulk-операций
- Поведение `staff`-роли для тикетов требует отдельного решения перед реализацией

## Критерии успеха

- [ ] Недопустимые роли получают 403 или редирект (не 500)
- [ ] Успешный create/update — редирект на index с flash
- [ ] Невалидные данные — повторный рендер формы с ошибками
- [ ] Удаление отеля с ассоциациями — flash-ошибка вместо исключения
- [ ] Request specs покрывают: success, validation failure, каждую роль
